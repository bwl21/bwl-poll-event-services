## ⚠️ Issues Found

### 🔴 HIGH Priority

#### 1. Shared Vue Instance - Potential State Leakage
**Location:** `src/main.ts` (lines 45-56)

Components are registered globally, creating shared state. When AdminPanel loads lazily, it re-uses the same component registry.

**Risk:** If multiple users access the app in same browser context, state could leak.

**Recommendation:**
```typescript
// Consider keeping admin components local to AdminPanel.vue
// Instead of global registration for components only used in admin
const AdminPanel = defineAsyncComponent(() => 
  import('./components/AdminPanel.vue')
)
// AdminPanel.vue registers Dialog, DataTable, etc. locally
```

#### 2. XLSX Vulnerability - Known Risk
**Version:** 0.18.5  
**Vulnerabilities:** Prototype Pollution + ReDoS (HIGH severity)  
**Status:** No patched version available  

**Impact:** 
- Affects Excel export only
- Exploitable if attacker controls ChurchTools data
- Risk is Medium (ChurchTools is trusted source)

**Recommendation:**
```typescript
// Document the known risk
// Consider migration to 'exceljs' (maintained, no known vulns)
// Timeline: Plan migration within 6 months
```

#### 2. Missing Error Boundaries in Components
**Location:** All components  

If an API call fails, errors bubble up. Components should handle gracefully.

**Example Issue:**
```vue
<!-- AdminConfig.vue -->
<!-- If loadConfigs() fails, no retry mechanism -->
<ProgressSpinner v-if="loading" />
<div v-else-if="configs.length === 0" class="empty-state">
  <!-- What if it's an error, not empty? -->
</div>
```

**Recommendation:**
```vue
<script setup>
const error = ref<Error | null>(null);

async function loadConfigs() {
  error.value = null;
  try {
    // ...
  } catch (e) {
    error.value = e as Error;
  }
}
</script>

<template>
  <div v-if="error" class="error-state">
    <p>{{ error.message }}</p>
    <Button @click="loadConfigs">Retry</Button>
  </div>
  <!-- ... -->
</template>
```

#### 3. Race Condition in Service Config Updates
**Location:** `AdminConfig.vue:79-102`

Multiple rapid toggles could cause race conditions.

**Current Code:**
```typescript
async function handleToggleVotes(...) {
    savingServiceId.value = config.serviceId;
    try {
        await updateServiceConfig(...);
        config.votesVisible = newValue; // Updates immediately
    } finally {
        savingServiceId.value = null;
    }
}
```

**Problem:** If second toggle happens while first is saving, both updates fire.

**Recommendation:**
```typescript
const isSaving = ref(new Set<number>());

async function handleToggleVotes(config, newValue) {
    if (isSaving.value.has(config.serviceId)) return; // Prevent double-tap
    isSaving.value.add(config.serviceId);
    try {
        const result = await updateServiceConfig(...);
        config.votesVisible = result.votesVisible; // Use server response
    } finally {
        isSaving.value.delete(config.serviceId);
    }
}
```
